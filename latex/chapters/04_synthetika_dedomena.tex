\chapter{Επέκταση με Συνθετικά Δεδομένα}

Το κεφάλαιο αυτό περιγράφει τον τρόπο παραγωγής συνθετικών ηχητικών
δειγμάτων (audio spoof) μέσω μηχανών Text-to-Speech (TTS) και την ενσωμάτωσή
τους στα derived manifests του project.
Η επέκταση αυτή αποτελεί τον πυρήνα του Phase~6+ και επιτρέπει στο σύστημα
να διακρίνει πραγματική ανθρώπινη ομιλία από TTS-παραγόμενη ομιλία.

% ============================================================
\section{Γιατί Χρειαζόμαστε Συνθετικά Audio}
\label{sec:why-tts}
% ============================================================

Το υποσύνολο MAVOS-DD που χρησιμοποιείται σε αυτό το project αποτελείται
κατά κύριο λόγο από \emph{visual-fake} πηγές: οι γεννήτριες EchoMimic,
MEMO, LivePortrait και Sonic παράγουν βίντεο στα οποία \textbf{το πρόσωπο
είναι συνθετικό, ενώ ο ήχος παραμένει πραγματικός} — δηλαδή, κανονικά
driving audio track από το αρχικό YouTube clip.
Στα derived manifests, όλα αυτά τα δείγματα φέρουν ετικέτα
\codeid{audio\_label = "bonafide"} ακόμη κι αν η εικόνα είναι fake.

Αυτή η δομή σημαίνει ότι δεν υπάρχει εγγενής διαφορά μεταξύ των
audio features bonafide και spoof στο βίντεο subset — ο μόνος τρόπος
για να εκπαιδευτεί audio anti-spoof κεφαλή είναι να εισαχθούν
\emph{πραγματικά TTS-παραγόμενα} audio δείγματα που φέρουν
\codeid{audio\_label = "spoof"}.

Επιπλέον, η multi-engine προσέγγιση εξυπηρετεί ένα αποκρυπτογραφικό
στόχο: αν το μοντέλο εκπαιδευτεί μόνο σε μία TTS μηχανή, κινδυνεύει
να μάθει \emph{spectral fingerprinting} — να αναγνωρίζει τις ιδιαίτερες
αρτεφάκτες codec/vocoder της συγκεκριμένης μηχανής αντί να μαθαίνει
γενικεύσιμα χαρακτηριστικά ομιλίας.
Η χρήση τριών ανεξάρτητων TTS providers (\engineName{ElevenLabs},
\engineName{Google Neural2}, \engineName{OpenAI TTS-1}) ενισχύει τη
διαφορετικότητα των spoof δειγμάτων και επιτρέπει ανάλυση per-provider
recall στα τελικά αποτελέσματα.

% ============================================================
\section{Pipeline: STT → TTS}
\label{sec:stt-tts-pipeline}
% ============================================================

Η παραγωγή συνθετικού audio ακολουθεί διπλό στάδιο: πρώτα μεταγράφεται
ο ήχος των πραγματικών βίντεο μέσω Speech-to-Text, και στη συνέχεια
παράγεται νέο συνθετικό audio από τα κείμενα αυτά μέσω TTS.
Κάθε παραγόμενο αρχείο \textbf{κληρονομεί το split του source video}
(train/val/test) αυστηρά — δεν υπάρχει διαρροή transcript/content
μεταξύ splits.

% ------------------------------------------------------------
\subsection{Transcription με Google STT V2}
\label{subsec:google-stt}
% ------------------------------------------------------------

Το script \codeid{scripts/transcribe\_google\_stt\_v2.py} χρησιμοποιεί
το Google Cloud Speech-to-Text API V2 για τη μαζική μεταγραφή των
\codeid{.wav} αρχείων από τον κατάλογο \codeid{data/audio\_wav/}.
Τα JSON αποτελέσματα αποθηκεύονται στον κατάλογο
\codeid{data/transcripts/google\_stt\_v2/}, ένα \codeid{.json} ανά βίντεο,
με ιεραρχία \codeid{<source\_folder>/<video\_id>.json}.

Κάθε JSON αρχείο περιέχει τα πεδία:
\codeid{audio\_path}, \codeid{source\_folder}, \codeid{video\_id},
\codeid{recognizer}, \codeid{language\_codes}, \codeid{model},
\codeid{transcript} (συνενωμένο κείμενο), \codeid{segments}
(confidence ανά τμήμα) και \codeid{response} (πλήρης API απάντηση).

Σημαντική λεπτομέρεια υλοποίησης: το πεδίο \codeid{model} στο Google
STT V2 API απαιτεί μη κενή τιμή — αντίθετα με το V1 που δέχεται
κενό string.
Η μεταγραφή εκτελείται \textbf{αποκλειστικά σε real-source WAV αρχεία}
(source folder \codeid{real}), δεδομένου ότι στόχος είναι η δημιουργία
TTS spoof από transcript πραγματικής ανθρώπινης ομιλίας.
Η μεταγραφή visual-fake βίντεο (EchoMimic, MEMO κ.λπ.) θα ήταν
περιττή αφού το audio τους είναι ήδη πραγματικό.

Το script υποστηρίζει idempotency: υπάρχοντα \codeid{.json} παραλείπονται
εκτός αν περαστεί η σημαία \codeid{--overwrite}.
Η επαλήθευση κόστους πριν από την κλήση API γίνεται με
\codeid{--estimate-only}, το οποίο αναφέρει τον αριθμό αρχείων,
τη διάρκεια σε λεπτά και την εκτιμώμενη χρέωση.
Μετά την εκτέλεση, ένα αρχείο \codeid{transcripts.jsonl} ενοποιεί
όλα τα επιμέρους JSON σε μία γραμμή ανά βίντεο.

% ------------------------------------------------------------
\subsection{Συντεταγμένη Παραγωγή Spoof Audio}
\label{subsec:tts-production}
% ------------------------------------------------------------

Κάθε TTS script διαβάζει τα JSON transcripts από
\codeid{data/transcripts/google\_stt\_v2/} και παράγει audio αρχεία
σε \textbf{ξεχωριστό υποκατάλογο ανά μηχανή}:

\begin{center}
\begin{tabular}{ll}
  \toprule
  Engine & Κατάλογος εξόδου \\
  \midrule
  \engineName{ElevenLabs}    & \codeid{data/tts\_audio/elevenlabs/<source\_folder>/} \\
  \engineName{Google Neural2} & \codeid{data/tts\_audio/google\_tts/<source\_folder>/} \\
  \engineName{OpenAI TTS-1}  & \codeid{data/tts\_audio/openai\_tts/<source\_folder>/} \\
  \bottomrule
\end{tabular}
\end{center}

Η ονομασία κάθε αρχείου ακολουθεί την κανονική σύμβαση:
\codeid{\{video\_id\}\_\_voice-\{voice\_id\_or\_name\}.\{ext\}}.
Η σύμβαση αυτή επιτρέπει στον builder manifests
(\codeid{src/data/build\_speech\_manifests.py}) να αναλύει (\emph{parse})
αυτόματα τον αναγνωριστή φωνής από το όνομα αρχείου μέσω του
regular expression \codeid{\_\_voice-}.

Και τα τρία scripts ακολουθούν το ίδιο idempotency pattern:
αν υπάρχει ήδη αρχείο που ταιριάζει στο pattern
\codeid{\{video\_id\}\_\_voice-*.ext} για ένα δεδομένο βίντεο, το δείγμα
παραλείπεται χωρίς κλήση API.
Αυτό επιτρέπει επανεκκίνηση από οποιοδήποτε σημείο διακοπής
χωρίς επαναχρέωση.
Η σημαία \codeid{--overwrite} αναγκάζει επανεγγραφή.

% ------------------------------------------------------------
\subsection{ElevenLabs Multilingual v2}
\label{subsec:elevenlabs}
% ------------------------------------------------------------

Το script \codeid{scripts/synthesize\_tts\_from\_transcripts.py} καλεί
το ElevenLabs REST API (\codeid{/v1/text-to-speech/\{voice\_id\}})
χρησιμοποιώντας το μοντέλο \codeid{eleven\_multilingual\_v2}.

\paragraph{Παράμετροι.}
Η έξοδος είναι \textbf{MP3 44,1\,kHz / 128\,kbps}
(\codeid{output\_format = mp3\_44100\_128}).
Οι παράμετροι ποιότητας φωνής ορίζονται ως
\codeid{stability = 0.5} και \codeid{similarity\_boost = 0.75}.

\paragraph{Pool φωνών.}
Στο πραγματικό run χρησιμοποιήθηκαν \textbf{19 φωνές}
(\codeid{DEFAULT\_ELEVENLABS\_VOICE\_IDS} στον κώδικα),
οι οποίες αντιστοιχίζονται σε κάθε transcript με round-robin
βάσει της ταξινομημένης σειράς των αρχείων:
\codeid{voice\_id = voice\_ids[record\_index \% len(voice\_ids)]}.
Αυτό εξασφαλίζει ντετερμινισμό και ισοκατανομή φωνών.

\paragraph{Ανθεκτικότητα σε σφάλματα.}
Αν μία φωνή απαιτεί \codeid{paid\_plan} που δεν διατίθεται,
η script αυτόματα αναθέτει την επόμενη διαθέσιμη φωνή
(voice rotation) και καταγράφει τις απενεργοποιημένες φωνές.
Σε περίπτωση quota exhaustion (HTTP 429 ή \codeid{"quota"} στο
μήνυμα σφάλματος), η εκτέλεση διακόπτεται καθαρά με
\codeid{stopped\_for\_quota = True} — χωρίς silent loss.

\paragraph{Αρχείο manifest.}
Κάθε επιτυχημένη σύνθεση καταγράφεται στο
\codeid{data/tts\_audio/manifest.jsonl} με πεδία:
\codeid{source\_folder}, \codeid{video\_id}, \codeid{provider},
\codeid{voice\_id}, \codeid{model\_id}, \codeid{output\_format},
\codeid{text\_chars}, \codeid{transcript\_path},
\codeid{original\_audio\_path} και \codeid{synthetic\_audio\_path}.
Το manifest περιέχει \todoMetric{αριθμός ElevenLabs γραμμών manifest} γραμμές.

% ------------------------------------------------------------
\subsection{Google Neural2}
\label{subsec:google-tts}
% ------------------------------------------------------------

Το script \codeid{scripts/synthesize\_google\_tts\_from\_transcripts.py}
χρησιμοποιεί τη βιβλιοθήκη \codeid{google-cloud-texttospeech} μέσω
\codeid{TextToSpeechClient}.

\paragraph{Παράμετροι.}
Η έξοδος είναι \textbf{MP3 24\,kHz / 64\,kbps}
(\codeid{AudioEncoding.MP3} με default \codeid{speaking\_rate = 1.0}
και \codeid{pitch = 0.0}).
Το encoding αυτό διαφέρει σκόπιμα από το ElevenLabs (44,1\,kHz / 128\,kbps),
ένα παράγοντα που αντιμετωπίζεται αργότερα από το codec-matching
pipeline (βλ.\ Κεφάλαιο~7).

\paragraph{Pool φωνών.}
Χρησιμοποιήθηκαν \textbf{9 φωνές} της σειράς \codeid{en-US-Neural2}
(A, C, D, E, F, G, H, I, J) σε round-robin ανά transcript.
Εναλλακτικά, η παράμετρος \codeid{--voice-name} επιτρέπει override
του pool.

\paragraph{Αρχείο manifest.}
Τα αποτελέσματα αποθηκεύονται στο
\codeid{data/tts\_audio/google\_tts\_manifest.jsonl}.
Το manifest αυτό περιέχει \textbf{498 γραμμές} (επαληθεύεται από
το filesystem).

% ------------------------------------------------------------
\subsection{OpenAI TTS-1}
\label{subsec:openai-tts}
% ------------------------------------------------------------

Το script \codeid{scripts/synthesize\_openai\_tts\_from\_transcripts.py}
χρησιμοποιεί το Python SDK \codeid{openai>=1.0.0} με streaming response
(\codeid{with\_streaming\_response.create}) για άμεση εγγραφή στο δίσκο
χωρίς ενδιάμεση μνήμη.

Ο \engineName{OpenAI TTS-1} προστέθηκε στο Phase~6+ ως τρίτη TTS μηχανή,
με στόχο τη δοκιμή της \emph{engine-fingerprint generalization story}:
αν το μοντέλο εκπαιδευτεί σε ElevenLabs + Google και εφαρμοστεί σε
OpenAI, το per-provider recall για \engineName{OpenAI TTS-1} αποκαλύπτει
πόσο γενικεύσιμα είναι τα μαθηματά χαρακτηριστικά.

\paragraph{Παράμετροι.}
Η προεπιλεγμένη μορφή εξόδου είναι \textbf{WAV}
(\codeid{response\_format = "wav"}), σε αντίθεση με τις άλλες δύο
μηχανές που παράγουν MP3.
Αυτό σημαίνει ότι τα OpenAI δείγματα έχουν \emph{καθαρό WAV codec}
— ακριβώς όπως τα bonafide αρχεία.
Για αυτό, το codec-matching pipeline τα αντιμετωπίζει ξεχωριστά
μέσω \codeid{CLEAN\_WAV\_PROVIDERS = \{"openai\_tts"\}}:
εφαρμόζεται η ίδια MP3 round-trip κανονικοποίηση ώστε να μη
δημιουργηθεί αντίστροφη codec συσχέτιση (clean WAV = OpenAI spoof).

Το μοντέλο είναι \codeid{tts-1} (default) με υποστήριξη για
\codeid{tts-1-hd} και \codeid{gpt-4o-mini-tts} μέσω \codeid{--model}.

\paragraph{Pool φωνών.}
Χρησιμοποιήθηκαν \textbf{6 φωνές} (alloy, echo, fable, onyx, nova,
shimmer) σε round-robin.

\paragraph{Αρχείο manifest.}
Τα αποτελέσματα αποθηκεύονται στο
\codeid{data/tts\_audio/openai\_tts\_manifest.jsonl}.
Το manifest αυτό περιέχει \textbf{1047 γραμμές} (επαληθεύεται από
το filesystem).

\section{Γιατί η Coqui XTTS-v2 Δεν Συμπεριλήφθηκε}
\label{sec:coqui-dropped}
% ============================================================

Σχεδιάστηκε αρχικά να χρησιμοποιηθεί και τέταρτη TTS μηχανή,
η \engineName{Coqui XTTS-v2} — ένα open-source, τοπικά εκτελούμενο
multilingual zero-shot voice-cloning μοντέλο.
Το αντίστοιχο script (\codeid{scripts/synthesize\_coqui\_xtts\_from\_transcripts.py})
υλοποιήθηκε πλήρως, χρησιμοποιώντας τα 33 built-in studio voices του
XTTS-v2 σε round-robin, με native WAV 24\,kHz έξοδο.

Ωστόσο, η \engineName{Coqui XTTS-v2} εξαιρέθηκε από το τελικό pipeline
για τεχνικό λόγο dependency conflict:

\begin{itemize}
  \item Η βιβλιοθήκη \codeid{TTS>=0.22.0} (Coqui) απαιτεί
    \textbf{\codeid{numpy==1.22.0}} ως hard pin — μια έκδοση
    τριών ετών που δεν μπορεί να αναβαθμιστεί.
  \item Το υπόλοιπο περιβάλλον (librosa, matplotlib, pandas,
    scikit-learn, PyTorch κ.λπ.) απαιτεί \codeid{numpy>=1.23}
    ή νεότερο.
  \item Δεν υπάρχει τρόπος να συνυπάρξουν σε ένα \codeid{venv}:
    το \codeid{pip} αρνείται να εγκαταστήσει και τα δύο.
\end{itemize}

Ο μόνος τρόπος ενσωμάτωσης θα ήταν ξεχωριστό conda environment
με Python 3.9--3.11 και manual isolation του Coqui process.
Δεδομένου ότι η \engineName{Coqui XTTS-v2} ήταν nice-to-have
(τέταρτη μηχανή για πρόσθετη διαφορετικότητα) και όχι required
για το core engine-fingerprint experiment
(\engineName{ElevenLabs} + \engineName{Google Neural2} + \engineName{OpenAI TTS-1}
καλύπτουν ήδη τρεις ανεξάρτητες paid API μηχανές),
αποφασίστηκε να εξαιρεθεί από το τρέχον scope.

\section{Παραγωγή Manifests από TTS Output}
\label{sec:tts-manifests}
% ============================================================

Αφού παραχθούν τα TTS audio αρχεία, ενσωματώνονται στα derived manifests
του project μέσω του \codeid{src/data/build\_speech\_manifests.py}.

% ------------------------------------------------------------
\subsection*{Provider Registration}
% ------------------------------------------------------------

Η καταχώρηση του \codeid{openai\_tts} provider (PR~\#15) ήταν αναγκαία
επέκταση: αρχικά ο builder υποστήριζε μόνο τους providers
\codeid{elevenlabs} και \codeid{google\_tts}.
Παράλληλα, ο filename parser (\codeid{\_FILENAME\_RE}) επεκτάθηκε
για να αποδέχεται \codeid{.wav} extensions επιπλέον του \codeid{.mp3}.

Χωρίς αυτή την αλλαγή, τα OpenAI WAV αρχεία θα αγνοούνταν εντελώς
από τον filesystem scanner.

% ------------------------------------------------------------
\subsection*{Split Inheritance και Spoof Label}
% ------------------------------------------------------------

Η συνάρτηση \codeid{iter\_generated\_rows()} κατασκευάζει μία γραμμή
spoof ανά TTS record.
Κάθε γραμμή \textbf{κληρονομεί το split από το source video}.

Αν ένα source video ανήκει στο train split, τότε \emph{όλα} τα
TTS-παραγόμενα δείγματα από αυτό το video ανατίθενται αυτόματα
στο train split.
Δεν γίνεται καμία ανακατανομή transcript-level: δεν υπάρχει
κίνδυνος \emph{content leakage} (ίδιο κείμενο σε train και val).

Κάθε παραγόμενη γραμμή φέρει:
\begin{itemize}
  \item \codeid{audio\_label = "spoof"}, \codeid{audio\_label\_binary = 1}
  \item \codeid{video\_label = "na"} (δεν υπάρχει βίντεο)
  \item \codeid{sample\_id} = \codeid{\{provider\}\_\_\{video\_id\}\_\_voice-\{voice\}}
\end{itemize}

% ------------------------------------------------------------
\subsection*{Union JSONL και Filesystem Scan}
% ------------------------------------------------------------

Η \codeid{iter\_tts\_records()} υλοποιεί \emph{union} μεταξύ:
\begin{enumerate}
  \item Του \codeid{*.jsonl} manifest ανά engine (πλούσια metadata,
    εγγυημένα συγχρονισμένα με το run που τα δημιούργησε).
  \item Filesystem scan του υποκαταλόγου engine
    (\codeid{rglob("*")}, φιλτράρισμα \codeid{.mp3}|\codeid{.wav}).
\end{enumerate}

Αυτή η ένωση αντιμετωπίζει το φαινόμενο των \emph{stale manifests}:
αν ένα JSONL αρχείο είναι παλιό (ή ατελές λόγω interruption),
ο filesystem scan ανακαλύπτει αρχεία που το JSONL χάνει,
και τα ενσωματώνει στα derived manifests.
Σε περίπτωση collision (ίδιο \codeid{(provider, video\_id, voice)}),
το JSONL record έχει προτεραιότητα.

Κατά τη διάρκεια Phase~6+, χρειάστηκε rebuild των derived manifests
μετά από εκτέλεση του OpenAI TTS script, επειδή το αρχικό
\codeid{openai\_tts\_manifest.jsonl} είχε παραχθεί με παλαιότερη
έκδοση του script (πριν την καταχώρηση του provider).
Ο filesystem scan εξασφάλισε ότι όλα τα παραγόμενα \codeid{.wav}
αρχεία εντοπίστηκαν και συμπεριλήφθηκαν ορθά.
